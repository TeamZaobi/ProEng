# 外部 prompt 到 ProductEngineer 的能力映射

## 总原则

不要把外部 prompt 原封不动塞进 `SKILL.md`。
先判断它属于：

1. `core mode`
   - 直接成为 `ProductEngineer` 的主能力
2. `method`
   - 只吸收其工作法或判断准则
3. `external route`
   - 明确不由 `ProductEngineer` 接管

去过拟合规则：

- 只有会稳定提升主模式质量的内容，才允许进 `SKILL.md`
- 只有会稳定提升产品文档质量的内容，才允许变成运行时硬约束
- 只改善表达方式或协作手感的内容，优先做通用写作准则，不单独建层
- 只对维护者有价值的内容，留在 `references/`，不要成为运行时主说明
- 不只吸收章节标题，也吸收“什么才算好文档”的质量门槛

## 映射结果

### `PRD`

- 归类：`core mode`
- 落点：`prd`
- 吸收内容：
  - `prd.md` 产物约定
  - 章节大纲只能作为脚手架，不能成为固定写作顺序
  - `人性根因 -> 痛点识别 -> 需求覆盖 -> 产品定义 -> 组件拓扑 -> 用户故事` 的 PRD 生成顺序
  - 使用者熵减必须落成场景、done line 或验收信号
  - 组件拓扑必须早于用户故事，用户故事必须回指需求覆盖和验收
  - 用户故事唯一 ID
  - 可测试验收标准
  - 认证或授权故事的最低覆盖要求
  - 文档必须支撑范围与需求讨论，而不是模板填空
  - 去标识化标杆 PRD 的 `pain -> req -> component -> story` 覆盖矩阵和质量门槛
- 不吸收内容：
  - 任何执行任务或实现动作
  - 私有项目名、组件名、路径、ID、领域对象和内部 owner 语言

### `计划`

- 归类：`core mode`
- 落点：`plan`
- 吸收内容：
  - `plan.md` 产物约定
  - 计划依赖 `PRD`
  - 顺序化开发阶段
  - 前后端双侧任务覆盖
  - 测试、文档、部署、维护的补齐
  - 计划必须绑定真实功能、依赖和阶段退出条件
- 不吸收内容：
  - “计划代理替用户执行任务”的暗示

### `架构师`

- 归类：`core mode`
- 落点：`architecture`
- 吸收内容：
  - 分阶段分析
  - 90% 置信度门槛
  - 多方案比较
  - 模块、接口、数据、风险和技术成功标准
- 不吸收内容：
  - 直接越过澄清进入编码

### `架构评审`

- 归类：`method`
- 落点：`architecture`
- 吸收内容：
  - 模块切分不应机械映射需求或团队
  - 预判演进方向但避免过度设计
  - 模块切分要建立在需求内在逻辑关联上

### `教学`

- 归类：`not absorbed`
- 落点：不单独建层
- 吸收内容：
  - 只保留“先解释为什么”的通用表达倾向
- 不吸收内容：
  - 单独的 teaching 模式
  - 额外教学工作流
  - 把一般编程教学扩张成全栈授课代理

### `重构`

- 归类：`core mode`
- 落点：`refactor proposal`
- 吸收内容：
  - 先确认目标
  - 保持行为不变
  - 专注结构、可读性和可维护性
  - 为每条建议解释原因
- 不吸收内容：
  - 实现级大改写本身

### `AST+graph 2`

- 归类：`method`
- 落点：`refactor proposal`
- 吸收内容：
  - 在大型代码库里用结构化证据支撑重构建议
  - 用依赖图、调用图、AST 信息定位高耦合和复杂节点
  - 先分析再改
- 不吸收内容：
  - 把 `ProductEngineer` 定义成图数据库重构执行器

### `Vibe 编码`

- 归类：`not absorbed`
- 落点：不单独建层
- 吸收内容：
  - 无
- 不吸收内容：
  - 独立的 vibe 协作层
  - 视觉参考驱动的专门工作流
  - “直接帮用户把整个应用写完”的执行承诺

### `审计`

- 归类：`method + external route`
- 落点：
  - `domain audit` 吸收需求和实现覆盖审计的视角
  - exploit 级代码漏洞审计外部路由
- 吸收内容：
  - 认证授权
  - 输入验证
  - 数据保护
  - API 安全
  - Web 风险
  - 配置与依赖
  - DevOps 与 CI/CD 覆盖
- 不吸收内容：
  - 企业级漏洞扫描代理
  - `security-report.md` 代码审计产物

### `内容`

- 归类：`method + external route`
- 落点：
  - 写作风格被吸收到产出规范
  - 内容营销交付显式外部路由
- 吸收内容：
  - 直接
  - 信息密
  - 低废话
  - 面向外行也能读懂
  - 默认给出高质量第一稿，而不是把写作质量误解为追问数量
- 不吸收内容：
  - SEO 提纲
  - 营销文案
  - AI 检测规避型内容写作

### `苏格拉底提问法`

- 归类：`method`
- 落点：默认思考梯度
- 吸收内容：
  - 先问目标，再问证据、约束、备选和 done line
  - 默认只问最少的一组问题
  - 用来澄清和校准，不用来制造流程感
- 不吸收内容：
  - 无限追问
  - 把每次对话都变成教学或辩论
