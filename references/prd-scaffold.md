# PRD 脚手架

本脚手架来自成熟项目 PRD 的去标识化结构抽取。复用的是产品链条、覆盖矩阵和闭环检查，不复用原项目的领域内容、产品名、组件名、路径、ID 或内部 owner 语言。

## 标杆 PRD 去标识化规则

把真实项目 PRD 转成可复用案例或脚手架时，默认按下面规则处理：

1. 将产品、项目、团队、仓库、runtime、组件和代号替换成角色化名称，例如 `[产品]`、`[用户终端]`、`[能力引擎]`、`[项目对象]`、`[受控服务]`。
2. 将领域案例替换成中性对象，例如 `[领域对象]`、`[输入材料]`、`[受控资源]`、`[成果包]`、`[复核信号]`。
3. 删除本机路径、仓库名、线程 ID、项目 ID、组织名、私有数据源名和内部部署路线。
4. 保留产品生成链、覆盖矩阵和用户故事形状；这些才是可复用资产。
5. 不保留实施计划、运行时架构、测试命令、发布 gate 或治理 owner 裁决，除非它们已经直接表现为用户可见的产品约束。

## 推荐结构

```markdown
# PRD: [产品] 开发总需求

> role: `product_prd_truth`
> status: `[draft | active | final]`
> version: `[x.y.z]`
> last_updated: `[YYYY-MM-DD]`

## 1. 边界

本 PRD 只写产品需求链：

`人性根因 -> 痛点识别 -> 需求覆盖 -> 产品定义 -> 组件拓扑 -> 用户故事`

本 PRD 不定义运行时架构、实施任务、发布流程、仓库真源或项目写权。这些内容只有在影响用户需求或验收时，才作为产品约束被引用。

修改时优先把新判断融合进现有链条；不能融合时，先正向明示缺失的产品判断；不要用例外、黑名单或执行规则替代产品定义。

## 2. 人性根因

| root_ref | 根因 | 定义 | 产品含义 |
| --- | --- | --- | --- |
| `ROOT-001` | 比较优势 | 用户希望获得高于同类、过往或当前条件的结果。 | [产品帮助用户追求的更高水平结果。] |
| `ROOT-002` | 生存压力 | 用户需要在时间、资源、质量、合规或竞争压力下降低风险。 | [产品帮助用户抵御或控制的压力。] |

产品第一价值：

[用一到两句话说明产品为用户提供的可感知熵减。]

## 3. 痛点识别

| pain_ref | 用户痛点 |
| --- | --- |
| `PAIN-001` | [由人性根因造成的具体摩擦、焦虑、风险、浪费或阻断。] |
| `PAIN-002` | [第二个具体痛点。] |
| `PAIN-003` | [下游执行、连续推进、复核或恢复痛点。] |

## 4. 需求覆盖

需求只覆盖痛点，不替代产品定义、组件拓扑、实施计划或团队分工。

| req_ref | 产品需求 | 覆盖痛点 |
| --- | --- | --- |
| `REQ-001` | [能解除某个痛点的能力或产品行为。] | `PAIN-001` |
| `REQ-002` | [能解除另一个痛点的能力或产品行为。] | `PAIN-001`, `PAIN-002` |
| `REQ-003` | [用于连续推进、恢复、证据、协作或回执的能力。] | `PAIN-003` |

覆盖检查：

- 每个痛点至少有一个需求覆盖。
- 每个需求必须回指至少一个痛点。
- 无法回指痛点的需求只是候选 feature，不是已接受 PRD 需求。

## 5. 产品定义

`[产品]` 是 [一句话定义产品类别和核心用户承诺]。

产品先识别 [用户意图 / 成果预期 / 当前状态]，再生成 [路径 / 对象 / 决策 / 服务 / 输出]，从而减少 [混乱 / 风险 / 重复劳动 / 决策成本]。

核心约束：

`[成果形式] + [目标水平] + [时间成本] + [人力成本] + [资源成本]`

产品交付给用户的不是“泛化建议”，而是一组持续演进的产品对象：

| 产品对象 | 用户看到什么 |
| --- | --- |
| [预期对象] | [用户如何看到目标结果、约束或当前条件。] |
| [当前状态画像] | [用户看到的差距、成熟度或 readiness 判断。] |
| [工作对象] | [承载状态、边界、下一步、回执和证据的持久对象。] |
| [受控请求] | [带有准入、风险和限制的请求或服务对象。] |
| [回执] | [放行、受限、阻断、缺材料、复核状态和下一步动作。] |
| [成果包] | [包含方法、证据、限制和下一步路径的可复核输出。] |

非目标：

- [本 PRD 不定义的内容。]
- [应由架构、计划、治理或实施资产承接的内容。]

## 6. 组件拓扑

### 6.1 组件职责

| 组件 | 产品职责 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| `[产品体系]` | [整体产品承诺。] | [用户主要目标。] | [路径、服务或成果包。] |
| `[用户终端]` | [用户侧连续交互界面。] | [用户选择、想法、材料、修正意见。] | [当前状态判断、下一步、状态、回执。] |
| `[能力引擎]` | [领域推理或产品智能。] | [用户意图、当前状态、约束。] | [结构化对象、升级路径、决策路径。] |
| `[项目对象]` | [持久生命周期对象。] | [已确认路径、用户决策、服务状态。] | [状态、边界、下一步、证据。] |
| `[受控服务]` | [受控服务或资源访问面。] | [目的、范围、授权、字段、用途边界。] | [放行 / 受限 / 阻断回执和限制。] |
| `[成果包]` | [面向用户的可复核输出。] | [工作对象、推理路径、服务回执、限制。] | [交付物、报告、计划、提交件或 handoff。] |

### 6.2 产品链路

`[入口] -> [预期选择] -> [当前状态判断] -> [结构化对象] -> [升级或决策路径] -> [用户取舍确认] -> [项目对象] -> [受控服务判断] -> [回执] -> [成果包] -> [状态回写]`

## 7. 用户故事

用户故事必须覆盖第 4 部分的全部需求和第 6 部分的全部组件。每条故事都展示用户与产品的实际使用过程，不把内部架构、实施任务或测试流程写成用户故事。

### 7.1 全局使用主线

1. 用户进入 `[用户终端]`，表达 [目标结果和约束]。
2. 产品生成 [当前状态判断]，让用户看到自己离目标差在哪里。
3. 用户提供 [想法、材料、请求或资源线索]。
4. `[能力引擎]` 将输入转成 [结构化产品对象] 和 [路径]。
5. 用户在看到取舍后选择现实可行路线。
6. 产品创建或更新 `[项目对象]`。
7. `[项目对象]` 把路径拆成下一步动作，并记录状态、边界、回执和证据。
8. 如果需要受控资源，`[受控服务]` 返回放行、受限、阻断或缺材料状态。
9. 产品返回回执和限制，不暴露内部路径或受保护原始材料。
10. `[成果包]` 合并对象、路径、服务结果、限制和下一步路线，并回写到 `[项目对象]`。

### 7.2 故事清单

| story_ref | 视角 | 用户与产品的使用过程 | 回指需求 | 回指组件 |
| --- | --- | --- | --- | --- |
| `US-001` | [初次进入的用户] | [用户表达目标和约束；产品判断预期是否超出当前条件。] | `REQ-001` | `[用户终端]` |
| `US-002` | [只有想法的用户] | [用户输入非结构化想法；产品将其转成可判断、可推进的结构化对象。] | `REQ-001`, `REQ-002` | `[用户终端]`, `[能力引擎]` |
| `US-003` | [已有成熟材料的用户] | [用户提供已有材料；产品定位缺口并生成补强路径。] | `REQ-002` | `[能力引擎]`, `[成果包]` |
| `US-004` | [资源受限的用户] | [用户看到保守、标准和高目标路线的取舍，并选择可执行路线。] | `REQ-003` | `[用户终端]`, `[能力引擎]` |
| `US-005` | [回到旧项目的用户] | [用户之后回来；产品恢复状态、决策、回执、限制和下一步，不要求用户重讲全部上下文。] | `REQ-003` | `[用户终端]`, `[项目对象]` |

## 8. PRD 质量门槛

- 人性根因是显式的。
- 每个痛点都映射到一个或多个需求。
- 每个需求都能回指痛点。
- 产品定义说清持久产品对象，而不只是页面或功能。
- 组件拓扑说明职责、输入、输出、链路和需求覆盖。
- 每条用户故事都映射到需求和组件。
- 全局使用主线不依赖内部架构或实施计划也能读懂。
- 非目标把 plan、architecture、governance、runtime 和 release 工作挡在 PRD 外，除非它们是产品约束。
```
